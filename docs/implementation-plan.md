# 选煤厂一体化管控平台实现方案

## 1. 先明确这份文档要做的是什么

从招标技术规格书内容看，这不是单一的 Web 管理后台，而是一套覆盖以下范围的综合平台：

- 基础平台：私有云、网络、安全、调度中心、机房、交互终端
- 数据平台：统一采集、统一存储、数据治理、标准字典、共享接口、可视化、统一管控
- 自动化平台：PLC/仪表/阀门/皮带秤/灰分仪/料位计/视频等设备接入
- 生产控制：集控、自动启停、浅槽/旋流器/浮选/压滤/装车等控制与闭环优化
- 管理应用：生产报表、生产分析、煤质报表、煤质分析、停送电审批、远程停送电
- 设备管理：设备台账、设备履历、润滑、检修、工单、备件、在线监测
- 能源管理：电耗、介耗、药耗、油耗统计分析
- 安全与报警：风险隐患、视频联动、电子围栏、报警记录、报警统计、分级推送
- 多端交互：PC、Pad、手机统一账号统一界面

所以如果你要“做个一样功能的”，正确做法不是一次性全抄，而是分阶段建设。

## 2. 建议的实现边界

建议你把项目拆成两部分：

- 软件平台部分：你现在可以主导开发
- 现场自动化与硬件接入部分：需要和 PLC 厂家、仪表厂家、视频厂家、网络设备厂家联调

如果你当前目标是先把“平台”做出来，建议第一期只覆盖软件平台，不自己承诺改造所有现场控制逻辑。

## 3. 推荐总体架构

采用四层架构。

### 3.1 设备接入层

负责把现场设备数据标准化接进来。

- PLC 接入：Modbus TCP、OPC UA、S7、厂家私有协议
- 仪表接入：料位计、液位计、灰分仪、皮带秤、振动温度传感器
- 视频接入：RTSP、GB28181、厂商 SDK
- 业务系统接入：化验系统、ERP、OA、门禁、调度电话、能耗表计

建议单独做 `iot-gateway` 服务，不让业务系统直接连 PLC。

### 3.2 数据平台层

这是整个平台的核心。

- 实时数据：时序库存 1s 级采样数据
- 业务数据：关系库存工单、报表、审批、台账、权限
- 消息总线：报警、联动、推送、异步任务
- 统一数据模型：设备、测点、报警、工艺段、班次、物料、煤质指标
- 统一接口层：对 PC、移动端、第三方系统提供 API

建议：

- 时序数据库：`TimescaleDB` 或 `InfluxDB`
- 关系数据库：`PostgreSQL`
- 缓存：`Redis`
- 消息队列：`RabbitMQ` 或 `Kafka`
- 对象存储：`MinIO`

### 3.3 平台服务层

把共性能力做成平台服务。

- 用户/角色/权限中心
- 组织架构与岗位权限
- 设备中心
- 测点中心
- 报警中心
- 工作流中心
- 报表中心
- 文件中心
- 视频联动中心
- 消息通知中心
- 审计日志中心

这层决定后期能不能扩展。

### 3.4 应用层

按业务拆模块，不要做成一个巨型单体页面。

- 生产监控
- 工艺流程图
- 生产报表/分析
- 煤质管理
- 设备管理
- 点巡检/维保工单
- 停送电审批
- 远程停送电
- 能源管理
- 安全管理
- 报警管理
- 视频监控联动
- 大屏驾驶舱
- 移动端

## 4. 推荐技术选型

如果你希望后面同时兼顾 Web、移动端、接口联调和快速开发，我建议走下面这套：

### 后端

- `Java 21 + Spring Boot`
- `Spring Cloud` 可选，不建议一开始就全微服务
- `MyBatis Plus` 或 `JPA`
- `Flowable` 处理审批流
- `MQTT/OPC UA SDK/Modbus` 做设备接入

原因：

- 国内工业项目更容易招聘和交接
- 权限、审批、报表、集成类项目成熟度高
- 和国产数据库、中间件适配相对容易

### 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Element Plus` 作为基础后台组件
- 大屏图表：`ECharts`
- 工艺流程图：`LogicFlow`、`AntV X6` 或 SVG 自绘

### 移动端

- 优先 `uni-app` 或 `Flutter`

如果你要和微信生态结合，优先 `uni-app`。如果你更在意交互质量和长期可维护性，优先 `Flutter`。

### 视频能力

- 基础查看：NVR/厂商平台对接
- Web 播放：`HLS/FLV/WebRTC`
- 智能分析：优先复用摄像头/NVR 自带算法，不要第一期自研算法平台

### 部署

- `Docker`
- `Kubernetes` 可后置
- `Nginx`
- `GitLab CI` 或 `Jenkins`

## 5. 第一阶段应该做什么

第一阶段不要碰“全厂自动控制闭环优化”，先做统一平台底座和高价值应用。

### 第一阶段 MVP

建议只做这 8 个模块：

1. 用户权限与组织架构
2. 设备台账与测点管理
3. 实时监控总览
4. 报警中心
5. 生产报表
6. 煤质报表
7. 设备工单/检修管理
8. 大屏驾驶舱和移动端只读版

这一阶段的目标：

- 先把数据“接进来、存起来、看得见、能追溯”
- 不先做复杂闭环控制
- 不先做高风险远程停送电执行

## 6. 第二阶段再做什么

第二阶段做强业务闭环。

- 停送电审批流
- 电子挂牌
- 视频联动报警
- 能耗分析
- 点巡检与维保计划
- 隐患排查和风险分级
- 生产分析与煤质分析
- 移动端工单处理

## 7. 第三阶段再做什么

第三阶段才建议做自动控制优化和智能算法。

- 自动启停车策略
- 密度闭环控制
- 药剂优化模型
- 压滤协同控制
- 设备预测性维护
- 视频智能识别深度联动
- 大语言模型问答助手

这些功能不难做演示，难的是现场可靠运行和责任边界，所以必须后置。

## 8. 核心数据模型建议

你后面建库时，最少要有这些核心实体：

- 组织、用户、角色、岗位
- 设备、设备分类、设备位置、设备履历
- 测点、测点类型、采集协议、采集频率
- 报警规则、报警记录、报警通知
- 工艺段、流程图节点、流程图连线
- 班次、班报、日报、月报
- 煤质样本、化验指标、化验结果
- 工单、检修计划、润滑计划、备件
- 审批单、操作票、停送电任务
- 能耗记录、电耗/介耗/药耗/油耗
- 视频点位、联动规则、录像索引
- 文件附件、导入任务、审计日志

## 9. 关键难点

这个项目真正难的不是写 CRUD，而是下面几件事：

- 现场协议不统一，很多设备接口要逐个对厂家
- 实时数据和业务数据不能混着存
- 报警要可配置，否则后面全是硬编码
- 权限必须细到岗位和设备范围
- 工艺流程图不能只画图，必须和测点、报警、视频点位关联
- 停送电属于高风险业务，审批流和控制闭锁必须严格分层
- 视频联动不能直接绑死摄像头品牌

## 10. 最现实的落地路线

建议按下面顺序开工：

1. 先定统一数据模型
2. 再做设备接入网关
3. 再做平台底座：用户、权限、设备、测点、报警
4. 再做实时监控首页和工艺流程图
5. 再做生产报表、煤质报表、设备工单
6. 再做移动端
7. 最后再上审批闭环、视频联动、智能分析

## 11. 如果现在就开始做，我建议的项目拆分

可以直接拆成以下仓库或模块：

- `platform-web`：PC 端前端
- `platform-mobile`：移动端
- `platform-api`：业务 API
- `platform-iot-gateway`：设备采集网关
- `platform-job`：统计、报表、同步任务
- `platform-workflow`：审批与流程服务
- `platform-screen`：驾驶舱/大屏

如果团队很小，也可以先做单体：

- `web-admin`
- `server`
- `iot-gateway`

## 12. 结论

要做成和文档接近的能力，正确路线是：

- 先把它定义为“工业一体化平台项目”
- 第一阶段只做软件平台和数据底座
- 高风险控制功能后置
- 通过统一设备模型、测点模型、报警模型、工作流模型来支撑后续扩展

如果直接从零开始，我建议你先做一个可上线的第一期：

- 实时监控
- 报警管理
- 设备管理
- 生产报表
- 煤质报表
- 移动端查看

这套做出来，已经是一个可演示、可试点、可继续扩展的平台雏形。

# 设备全生命周期管理一期方案

## 1. 目标重定义

如果当前最需要的是“设备全生命周期管理”，那这个项目的一期目标就不应该是整厂一体化总平台，而应该是：

- 先做设备资产与运维管理平台
- 为后续报警、在线监测、能耗、工艺联动预留接口
- 先解决设备台账不清、履历断档、维保靠纸面、故障闭环差、备件不好管的问题

这类项目本质上更接近：

- `EAM` 设备资产管理
- `CMMS` 设备维护管理
- 再叠加部分工业现场数据能力

## 2. 生命周期范围

建议把“全生命周期”定义为以下 8 个阶段：

1. 规划立项
2. 采购到货
3. 安装调试
4. 投运建档
5. 点检巡检
6. 维保检修
7. 故障维修
8. 报废替换

系统必须保证每个阶段都有记录、有责任人、有时间、有附件、有状态变更痕迹。

## 3. 一期必须做的模块

### 3.1 设备主数据中心

这是根基，优先级最高。

- 设备分类树
- 设备编码规则
- 设备台账
- 设备层级关系
- 设备位置结构
- 设备技术参数
- 设备厂家与供应商
- 设备铭牌信息
- 附件管理

关键要求：

- 一台设备只能有一个主档案
- 一个设备可以挂多份文档、图纸、说明书、照片
- 支持按厂区、车间、工段、系统、设备位号快速检索

### 3.2 设备履历

- 安装记录
- 调试记录
- 改造记录
- 故障记录
- 检修记录
- 更换配件记录
- 停机记录
- 启停历史
- 报废记录

这是“全生命周期”最核心的一张链路。

### 3.3 点检巡检

- 巡检路线
- 巡检项目模板
- 巡检周期
- 巡检任务下发
- 巡检结果录入
- 异常上报
- 图片/语音/附件上传
- 漏检统计

移动端必须支持，不然一线人员不会真正用起来。

### 3.4 维保计划

- 保养标准库
- 保养周期配置
- 维保计划自动生成
- 维保任务派发
- 维保执行反馈
- 维保验收
- 超期预警

建议按“时间周期 + 运行周期”双模式设计，后面才能接在线运行时长。

### 3.5 故障与工单管理

- 故障报修
- 工单派发
- 抢修处理
- 故障原因分析
- 停机时长统计
- 工单验收
- 工单关闭
- 故障知识库沉淀

这里必须把“现象、原因、措施、责任、耗时、耗材”结构化。

### 3.6 备件与材料管理

- 备件台账
- 备件分类
- 安全库存
- 领用归还
- 备件消耗记录
- 备件与设备关联
- 常用 BOM

没有这个模块，后面设备成本分析做不起来。

### 3.7 统计分析

- 设备完好率
- 开机率
- 停机率
- 故障次数
- 平均故障间隔时间 `MTBF`
- 平均修复时间 `MTTR`
- 维保达成率
- 巡检达成率
- 备件消耗排行
- 设备生命周期成本

管理层真正关心的是这些指标，不是录了多少表单。

## 4. 一期建议暂缓的功能

这些功能可以预留，但不建议一期重投入：

- 复杂审批流
- 远程停送电控制
- 自动控制闭环
- 视频智能识别
- 大模型助手
- 复杂三维可视化
- 全量能源管理

原因很简单：这些功能要么风险高，要么实施重，要么依赖现场改造，不适合用来打一期。

## 5. 一期系统架构建议

### 前端

- PC 管理端
- 移动端巡检/工单端

### 后端核心服务

- 用户权限服务
- 设备主数据服务
- 履历服务
- 巡检服务
- 维保服务
- 工单服务
- 备件服务
- 统计分析服务
- 附件与消息服务

### 数据层

- 关系库保存业务数据
- 对象存储保存图纸、照片、说明书、报告
- 缓存用于任务提醒和热点查询

如果一期先不做大规模实时采集，时序数据库可以先不引入。

## 6. 一期数据库核心实体

最少要有这些表：

- `asset_category` 设备分类
- `asset_location` 设备位置
- `asset_device` 设备台账
- `asset_device_relation` 设备层级关系
- `asset_supplier` 供应商
- `asset_file` 设备附件
- `asset_change_log` 生命周期变更记录
- `asset_event` 设备事件
- `inspection_template` 巡检模板
- `inspection_task` 巡检任务
- `inspection_record` 巡检记录
- `maintenance_template` 保养模板
- `maintenance_plan` 维保计划
- `maintenance_work_order` 维保工单
- `fault_report` 故障报修
- `repair_work_order` 维修工单
- `spare_part` 备件
- `spare_part_stock` 库存
- `spare_part_usage` 备件领用
- `device_runtime_stat` 设备运行统计

## 7. 设备生命周期状态机建议

设备状态不要只做一个字符串，建议做受控状态流转：

- 草稿
- 待验收
- 已投运
- 运行中
- 停机中
- 检修中
- 故障中
- 封存
- 已报废

每次状态变更都必须记录：

- 变更前状态
- 变更后状态
- 操作人
- 变更时间
- 原因
- 关联单据

## 8. 最重要的业务闭环

一期至少要打通 4 条闭环：

1. 建档闭环
设备新增 -> 验收 -> 投运 -> 台账生效

2. 巡检闭环
任务生成 -> 现场巡检 -> 异常上报 -> 生成工单 -> 工单关闭

3. 维保闭环
计划生成 -> 维保执行 -> 验收确认 -> 履历沉淀

4. 故障闭环
故障报修 -> 派单 -> 维修 -> 试车 -> 验收 -> 原因归档 -> 指标统计

## 9. 最现实的开发优先级

推荐按下面顺序开发：

1. 用户权限
2. 设备分类、位置、台账
3. 设备履历
4. 巡检管理
5. 故障报修与维修工单
6. 维保计划
7. 备件管理
8. 统计分析
9. 移动端

## 10. 你现在应该怎么做

如果目标明确是设备全生命周期管理，我建议立刻把项目范围收敛为一期：

- PC 端：设备台账 + 履历 + 维保 + 工单 + 统计
- 移动端：巡检 + 报修 + 工单处理 + 拍照上传

先不要把项目定义成“全厂智能化总平台”。
先把设备管理这条主线做深做透，后面再把报警、在线监测、能耗、视频逐步挂进来。

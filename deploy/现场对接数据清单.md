## 〇、与「智能停送电」现场方案对齐（必读）

现场已有一套**可运行的停送电系统**，其数据接入方式将作为本管控平台的**统一参考**。项目路径：

`D:\金海泽地\代码\PowerOutageManagement\智能停送电`

### 〇.1 停送电侧已验证的对接方式

| 项目 | 停送电（参考实现） | 管控平台（本仓库） |
|---|---|---|
| 现场 OPC | `OPCUA_ENDPOINT=opc.tcp://<KEPServer IP>:49320`（见 `docker-compose.yml`、`.env`） | `KEPSERVER_ENDPOINT`（`application.yml` → `platform.iot.kepserver.endpoint`） |
| 认证 | `OPCUA_USERNAME` / `OPCUA_PASSWORD`、`OPCUA_SECURITY_POLICY`（如 `None`） | `KEPSERVER_USERNAME` / `KEPSERVER_PASSWORD`（空则匿名） |
| 点位列表 | 环境变量 `OPCUA_NODEIDS`：逗号分隔的 **KEPServer 标识符字符串**（如 `AB.PLC1.AB.Online.刮板机带电`） | 数据库表 **`iot_tag`**：字段 **`source_path`** 存**同形态**的 OPC 标识符；`tag_code` 为平台内业务编码 |
| 命名空间 | `OPCUA_NS`（默认 `2`，与 KEPServer 常见配置一致） | 当前订阅代码固定 **`NodeId(2, sourcePath)`**（与停送电默认 NS=2 对齐；若现场 NS 不同需改 `OpcUaBootstrap.java`） |
| 数据源切换 | `DEVICE_DATA_SOURCE=opcua` / `mqtt` / `both` | 当前以 **OPC-UA → PostgreSQL `iot_realtime` / `iot_history`** 为主；MQTT 若需与停送电一致，需在 Spring 侧另行扩展 |
| 部署文档 | `DEPLOYMENT_GUIDE_SIMPLE.md`、`INSTALL.md`、`docker-compose.yml` | `deploy/部署指南.md`、本文件 |

**结论**：现场在停送电里已经调通的 **KEPServer 地址、账号、安全策略、NodeId 字符串写法**，在管控平台侧应**原样复用**到环境变量与 `iot_tag.source_path` 中，避免两套写法不一致。

### 〇.2 从停送电的 `OPCUA_NODEIDS` 迁移到管控平台

1. 打开停送电项目 `智能停送电/docker-compose.yml`（或现场 `.env`）中的 `OPCUA_NODEIDS`。
2. 将每个逗号分隔的标识符，在管控平台 **系统设置 → 点位台账**（或 SQL 写入 `iot_tag`）中新增一行：
   - **`source_path`** = 该标识符（与 `OPCUA_NODEIDS` 中单项完全一致）。
   - **`tag_code`** = 平台内唯一编码（建议小写点号风格，如 `scraper.powered`）。
   - **`source_type`** = `KEPSERVER`。
3. 设置 `KEPSERVER_ENABLED=true` 并重启后端，确认日志出现「OPC-UA 订阅启动完成」且 `iot_realtime` 有写入。

### 〇.3 参考文档索引（停送电仓库内）

| 文件 | 内容 |
|---|---|
| `DEPLOYMENT_GUIDE_SIMPLE.md` | 现场 `.env`、OPC/MQTT 切换、Docker 启动 |
| `INSTALL.md` | 安装与网络（含 `host.docker.internal`、OPC 证书说明） |
| `docker-compose.yml` | `OPCUA_*`、`MQTT_*`、`DEVICE_DATA_SOURCE` 默认值 |
| `REALTIME_STATUS_ARCHITECTURE.md` | 实时状态与持久化思路（停送电用 MySQL；管控平台用 PostgreSQL + `iot_realtime` 表，理念可对照） |

### 〇.4 去现场前可在办公室完成的步骤

1. **预置点位 SQL**：在测试库执行 `deploy/sql/seed-iot-tags.sql`，确认 `iot_tag` / `iot_tag_mapping` 有数据；现场改为真实 `source_path` 后再执行一次或走「系统设置」页面编辑。
2. **HTTP 路径**：后端已统一为 **`http://主机:8080/api/...`**（不再使用 `server.servlet.context-path` 叠一层 `/api`，避免 `/api/api/...`）。**Docker**：浏览器只访问 **`http://主机:WEB_PORT/`**（默认 80），由 **`web` 容器** 内 Nginx 把 **`/api/`** 转到 **`http://server:8080/api/`**，无需在浏览器里写 8080。
3. **WebSocket**：浏览器连接 **`ws://主机/ws/iot`**（与页面同源；Docker 下由 `web` 容器转到后端 **`/api/ws/iot`**）。OPC 有数据时由后端推送。
4. **OPC 命名空间**：与停送电 `OPCUA_NS` 对齐，默认 `2`；若现场不同，设置环境变量 **`KEPSERVER_OPCUA_NAMESPACE_INDEX`**（或 `application.yml` 中 `platform.iot.kepserver.opcua-namespace-index`）。

### 1.3 KEPServer 配置要点

- OPC-UA 端点地址：`opc.tcp://<服务器IP>:49320`
- 认证方式：匿名 / 用户名密码（需与 Spring Boot 配置一致）
- Spring Boot 配置项（`application.yml`）：

```yaml
kepserver:
  endpoint: opc.tcp://localhost:49320
  security-policy: None   # 或 Basic256Sha256
  username: ""
  password: ""
  scan-interval-ms: 1000
```

## 三、各页面数据对接状态

### 3.1 已对接后端 API 的页面（优先级低）

| 页面 | 已接的 API | 现场需做 |
|---|---|---|

| 页面 | 当前数据源 | 需要对接的数据 |
